Capítulo 19: Otimização de Consulta

Guia Completo e Aprofundado para Otimização em SGBDs

O Que É Otimização de Consulta?

Definição Simples: É o processo de fazer com que as consultas SQL executem o mais rápido possível, usando menos recursos do computador (memória, processador, disco).

Analogia do GPS

Imagine que você quer ir de casa ao trabalho. O GPS pode sugerir várias rotas diferentes - uma pela avenida principal, outra por ruas menores, uma terceira evitando trânsito. A otimização de consulta funciona como um "GPS inteligente" para dados: analisa todas as formas possíveis de buscar informações no banco de dados e escolhe a rota mais eficiente.

Por Que É Importante? Uma consulta mal otimizada pode demorar horas para executar, enquanto a mesma consulta otimizada pode executar em segundos. Em aplicações com milhões de usuários, isso significa a diferença entre um sistema que funciona bem e um que trava constantemente.

Conceito Fundamental: O termo "otimização" é tecnicamente incorreto - na verdade, buscamos uma solução muito boa, não necessariamente a perfeita, porque encontrar a solução perfeita poderia demorar mais que executar a consulta original!

119.1 Estruturas Básicas de Consulta

Árvores de Consulta - O Mapa da Execução

Exemplo Prático: Consulta de Funcionários

Imagine que você quer saber: "Quais funcionários do departamento de TI ganham mais de R$ 5.000?"

SELECT nome, salario
FROM funcionario, departamento
WHERE funcionario.dept_id = departamento.id
AND departamento.nome = 'TI'
AND funcionario.salario > 5000

O que acontece internamente:

  • O sistema precisa juntar duas tabelas (funcionario e departamento)
  • Filtrar apenas funcionários do departamento de TI
  • Filtrar apenas salários acima de R$ 5.000
  • Retornar apenas nome e salário

Árvore de Consulta

É como um fluxograma que mostra a ordem das operações:

  • Folhas (embaixo): As tabelas originais
  • Galhos (meio): Operações como junção, filtros
  • Raiz (topo): Resultado final

A árvore é executada de baixo para cima, como uma linha de produção.

O Problema da Explosão Combinatória

Cenário: Para uma consulta juntando apenas 4 tabelas, existem 24 ordens diferentes possíveis!

Para 10 tabelas: Mais de 3 milhões de possibilidades!

Solução: O otimizador usa técnicas inteligentes para examinar apenas as alternativas mais promissoras.

Regras Heurísticas - O Senso Comum da Otimização

Analogia da Cozinha

Imagine que você está fazendo um jantar para 10 pessoas. É mais eficiente:

  • Ruim: Cozinhar para 100 pessoas e depois jogar 90 pratos fora
  • Bom: Já começar cozinhando apenas para 10 pessoas

As regras heurísticas aplicam esse "senso comum" às consultas: fazem filtros primeiro para reduzir a quantidade de dados processados.

As 6 Regras de Ouro da Otimização Heurística

  1. Quebrar filtros complexos: Se você tem "WHERE idade > 25 AND salario > 3000", trate como dois filtros separados
  2. Aplicar filtros o mais cedo possível: Se você sabe que quer apenas funcionários de SP, filtre isso antes de fazer qualquer junção
  3. Reorganizar tabelas: Coloque as tabelas que vão ser mais filtradas primeiro na ordem de processamento
  4. Transformar produtos cartesianos em junções: Evite gerar milhões de combinações desnecessárias
  5. Selecionar apenas colunas necessárias: Se você só precisa de nome e salário, não carregue endereço, telefone, etc.
  6. Agrupar operações relacionadas: Execute operações que podem ser feitas juntas de uma vez só

Exemplo Real de Otimização

Consulta Original (RUIM):

1. Juntar TODAS as tabelas funcionario (10.000) e departamento (10)
2. Resultado: 100.000 combinações
3. Filtrar departamento = 'TI' → Sobram 1.000
4. Filtrar salario > 5000 → Sobram 50
Total processado: 100.000 registros

Consulta Otimizada (BOA):

1. Filtrar departamento = 'TI' primeiro → 1 registro
2. Filtrar funcionarios com salario > 5000 → 500 registros
3. Juntar apenas esses → 50 registros
Total processado: 551 registros (180x mais eficiente!)

Programação Dinâmica - Evitando Retrabalho

O Princípio da Programação Dinâmica

É como resolver um quebra-cabeças gigante dividindo em pedaços menores e lembrando das soluções que já encontrou para não ter que resolver a mesma parte duas vezes.

Exemplo Prático: 5 Tabelas

Sem otimização: 120 (5!) combinações possíveis para testar

Com programação dinâmica: Apenas 12 combinações precisam ser testadas

Economia: 90% menos processamento!

Como funciona:

  • Resolve primeiro as junções de 2 tabelas e guarda os resultados
  • Usa esses resultados para resolver junções de 3 tabelas
  • Continua assim até resolver toda a consulta
  • Nunca refaz um cálculo que já foi feito

219.2 Estratégias de Execução

Materialização vs Pipeline - Como Processar os Dados

Materialização (Como uma Fábrica Tradicional)

Processo:

  • Executa uma operação completamente
  • Salva resultado em arquivo temporário
  • Passa para próxima operação

Vantagens: Usa menos memória RAM

Desvantagens: Mais lento, usa muito disco

Pipeline (Como uma Linha de Montagem)

Processo:

  • Passa dados diretamente entre operações
  • Não cria arquivos intermediários
  • Processa tudo "ao vivo"

Vantagens: Muito mais rápido

Desvantagens: Usa mais memória RAM

Analogia da Lavanderia

Materialização: Lava toda a roupa, espera secar completamente, depois passa toda a roupa de uma vez.

Pipeline: Enquanto uma peça está secando, você já está passando outra que secou antes, e lavando uma terceira.

O pipeline é mais eficiente porque aproveita melhor o tempo e não desperdiça recursos.

Subconsultas - O Problema das Consultas Dentro de Consultas

Exemplo de Subconsulta Problemática

SELECT nome FROM funcionario
WHERE salario > (SELECT AVG(salario) FROM funcionario WHERE dept_id = funcionario.dept_id)

Problema: Para cada funcionário (10.000), executa a subconsulta interna!

Total de execuções: 10.001 consultas (1 externa + 10.000 internas)

Tempo estimado: 100x mais lento que necessário

Solução: Desaninhamento (Unnesting)

Transformar a subconsulta em uma junção normal:

SELECT f1.nome FROM funcionario f1
JOIN (SELECT dept_id, AVG(salario) as media FROM funcionario GROUP BY dept_id) f2
ON f1.dept_id = f2.dept_id
WHERE f1.salario > f2.media

Resultado: Apenas 2 operações principais em vez de 10.001!

Visões Materializadas - Resultados Pré-Calculados

Analogia do Relatório Mensal

Imagine que todo dia o chefe pede um relatório de vendas do mês. Você pode:

  • Opção 1 (Ruim): Todo dia recalcular todas as vendas do mês inteiro
  • Opção 2 (Boa): Manter uma planilha com o total do mês e só atualizar com as vendas do dia

Visões materializadas são como a "Opção 2" - mantêm resultados pré-calculados e só atualizam o que mudou.

Exemplo Real: Dashboard de Vendas

Consulta Frequente:

SELECT produto, SUM(quantidade), SUM(valor)
FROM vendas v JOIN produtos p ON v.produto_id = p.id
WHERE data >= '2024-01-01'
GROUP BY produto

Sem visão materializada: Toda vez que alguém abrir o dashboard, recalcula tudo (lento)

Com visão materializada: Resultado já está calculado, só precisa ler (rápido)

Estratégias de Atualização:

  • Imediata: Atualiza a cada venda (tempo real, mas pode ser lento)
  • Noturna: Atualiza uma vez por dia (rápido, mas dados podem estar desatualizados)
  • Sob demanda: Atualiza quando alguém pede o relatório

319.3 Estimativas de Custo

Os 5 Componentes de Custo

Analogia da Construção de uma Casa

Assim como construir uma casa tem vários custos (material, mão de obra, tempo), executar uma consulta também tem:

Componente O Que É Analogia Quando É Crítico
E/S de Disco Tempo para ler/escrever dados no HD/SSD Buscar documentos no arquivo Bancos grandes, dados não estão em RAM
CPU Tempo de processamento (ordenação, comparação) Tempo gasto calculando Operações complexas, dados em RAM
Memória Quantidade de RAM necessária Espaço na mesa de trabalho Consultas com muitas junções grandes
Armazenamento Espaço para resultados temporários Gavetas para guardar trabalho intermediário Consultas muito grandes
Rede Tempo para enviar dados pela internet Tempo de correio Bancos distribuídos, aplicações remotas

Estatísticas do Catálogo - As Informações que o BD Coleta

O que o Banco de Dados "Sabe" Sobre Seus Dados

Para escolher a melhor estratégia, o BD mantém estatísticas sobre as tabelas:

Sobre as Tabelas

  • Número de registros: Quantas linhas tem
  • Tamanho dos registros: Quanto espaço cada linha ocupa
  • Número de páginas: Como está organizado no disco

Sobre os Índices

  • Profundidade: Quantos níveis tem
  • Seletividade: Quão únicos são os valores
  • Distribuição: Como os valores estão espalhados

Sobre os Valores

  • Valores distintos: Quantos valores únicos existem
  • Valores mais comuns: Quais aparecem com mais frequência
  • Faixas de valores: Qual o mínimo e máximo
